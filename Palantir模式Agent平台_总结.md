# Palantir模式Agent平台(ERP方向)
## ERP智能体 — 技术方案总结

---

## 一、项目定位

**轻量化 Palantir 情报研判范式，落地 ERP 供应链智能分析**

| 维度 | 说明 |
|------|------|
| 核心能力 | 多实体自动抽取 → 语义检索 → 图谱关联 → 风险评分 → AI推演 → 实时可视化 |
| 技术路线 | Go + Langchaingo + DeepSeek + Chroma + BoltDB + D3.js |
| 部署方式 | 单二进制文件，Mac 一键启动，零外部依赖 |
| 数据规模 | 12类YonSuite实体，240条演示数据，全链路互联 |

## 二、双知识库架构（核心差异化能力）

本项目拥有两套知识库，各司其职、缺一不可：

### 2.1 向量知识库（Chroma — 语义检索层）

> 行业常说的 RAG 知识库，属于**语义文档知识库**

| 维度 | 说明 |
|------|------|
| 存储内容 | 每个 ERP 实体的完整描述文本 + 云端生成的 Embedding 向量 + 实体元数据 |
| 核心作用 | 用户自然语言提问不靠关键词匹配，靠语义相似度快速召回相关实体资料 |
| 防幻觉 | LLM 回答时读取原文，保证结论可溯源、不编造 |
| 归属 | RAG 检索、对话回答的底层知识库 |

### 2.2 图谱关系知识库（BoltDB — 关联链路层）

> 属于**结构化关系知识库**，是 Palantir 模式的灵魂

| 维度 | 说明 |
|------|------|
| 存储内容 | 业务关系三元组 `[实体A] —[关系]→ [实体B]`，如 `sup001 —[供货]→ mat001` |
| 核心作用 | 把向量库召回的孤立实体连成完整业务链路，实现多层深度溯源、上下游穿透分析 |
| 可视化 | D3 力导向图谱实时渲染完整业务网络，是右侧图谱的数据来源 |
| 归属 | 图谱遍历、关联溯源、可视化渲染的底层知识库 |

### 2.3 知识图谱的构建方式

知识图谱是**系统自动生成**与**人工定义**相结合的结果：

| 方式 | 说明 | 示例 |
|------|------|------|
| **自动生成** | 系统解析 ERP 结构化数据（合同、设备、供应商表）、日志、文档，自动抽取实体及其关系，并持续更新 | 读取 `purchase_order.json` 自动生成 `sup001 —[执行采购]→ po001` |
| **系统定义** | 图谱的本体模型（实体类型、关系类型、属性字段）由系统或领域专家预先定义，规范数据结构和语义 | 12类YonSuite实体类型 + 12种关系类型预先编码在 `bolt_graph.go` 中 |

### 2.4 双库对比：普通RAG vs 本平台

| 对比维度 | 普通单向量知识库（简单问答机器人） | 本平台 ERP 复合型知识库（Palantir） |
|----------|----------------------------------|--------------------------------------|
| 存储载体 | 仅向量库 | **Chroma 向量知识库 + BoltDB 图谱知识库** 双库联动 |
| 存储内容 | 纯文本段落 | 实体描述文本 + **实体关联关系** |
| 检索能力 | 仅语义匹配文档片段 | 1. 语义召回实体 2. **自动遍历多层业务链路** 3. AI推演下一步关联实体 |
| 可视化 | 无实体链路图谱 | **D3 实时渲染完整业务网络**，实线=已入库 / 黄色虚线=AI预判待核查 |
| 决策能力 | 无推理流程记录 | **完整6步AI决策日志**，展示检索深度、风险打分、执行步骤 |
| 溯源能力 | 只能引用文档片段 | 可沿图谱链路**多层穿透溯源**，精确定位风险根因 |

## 四、RAG 研判完整查询流程

```
用户提问
  │
  ▼
┌─────────────────────────────────────────────┐
│ Step 1: 意图识别                             │
│ LLM 识别业务域(供应链/采购/财务/库存)          │
│ + 自动设定检索 TopK、图谱遍历深度              │
└──────────────┬──────────────────────────────┘
               ▼
┌─────────────────────────────────────────────┐
│ Step 2: 向量召回 (Chroma)                    │
│ 用户问题 → 生成查询向量 → Chroma 语义匹配     │
│ → 召回相关 ERP 实体文档                      │
└──────────────┬──────────────────────────────┘
               ▼
┌─────────────────────────────────────────────┐
│ Step 3: 图谱深度遍历 (BoltDB)                 │
│ 以召回实体为起点 → 查询1~3度关联三元组        │
│ → 串联上下游完整业务链路                      │
└──────────────┬──────────────────────────────┘
               ▼
┌─────────────────────────────────────────────┐
│ Step 4: 风险评分 (多维打分引擎)               │
│ 7个维度(延期/库存/财务/合同/质检/设备/物流)    │
│ → 量化每实体风险分数，标记高风险节点           │
└──────────────┬──────────────────────────────┘
               ▼
┌─────────────────────────────────────────────┐
│ Step 5: 推演预判 (AI预测)                    │
│ AI 分析现有链路 → 预测下一步待核查实体         │
│ → 生成黄色虚线预判链路推送到图谱               │
└──────────────┬──────────────────────────────┘
               ▼
┌─────────────────────────────────────────────┐
│ Step 6: 生成研判报告 (LLM整合)                │
│ 整合向量文档 + 图谱关联链路 + 风险评分         │
│ → 输出带溯源、风险标记的结构化回答             │
└─────────────────────────────────────────────┘
```

每一步推理动作实时记录 **DecisionTrace**，通过 WebSocket 推送前端决策日志面板。

## 五、Palantir 四层核心业务执行链路

### 第一层：数据接入层（双模式实体生成入口）

两种数据来源，统一输出标准化 `ERPEntity` 实体结构：

| 模式 | 说明 | 流程 |
|------|------|------|
| API 自动模式 | 请求内置 Mock ERP 接口 | 获取 JSON → LLM 自动解析主键/外键/实体类型/关联关系 |
| 手动录入模式 | 页面表单输入实体描述 + 关联文本 | LLM 标准化生成实体 ID、三元组关系 |

### 第二层：双存储持久化层（向量 + 图谱分离存储）

| 分支 | 存储 | 流程 |
|------|------|------|
| 向量分支 | Chroma | 实体描述文本 → Embedding 向量 → 存入 Chroma，绑定元数据供语义检索 |
| 图谱分支 | BoltDB | 解析实体 Relations 关联数组 → 生成三元组 → 写入 BoltDB 实体图谱库 |

同步通过 WebSocket 推送新增实体节点、连线至前端 D3 图谱实时渲染。

### 第三层：AI 研判推理层（Langchaingo 核心，Palantir 决策引擎）

6步完整研判流程（详见第四章），每一步携带推理理由和 AI 下一步计划。

### 第四层：实时可视化展示层

三列布局，实时联动：
- **左列**：对话 Chat 窗口，Markdown 报告渲染（表格/列表/代码块/粗体）
- **中列**：D3 力导向图谱，分阶段增量展开 + 自动缩放适配
- **右列**：API 同步/手动录入面板 + AI 推理日志（显示当前步、AI 计划、多分支路径）

## 七、核心技术亮点

### 7.1 双知识库联动检索
- Chroma 做语义召回 → BoltDB 做链路深挖 → LLM 整合双源数据
- 不是简单文档问答，而是「语义理解 + 结构化推理」的复合智能

### 7.2 智能意图识别 + 自适应回答
- LLM 自动判断查询类型（列举/分析/风险）
- 自适应调整检索参数（TopK=3~20、遍历深度=1~3）
- 列举类问题 → 总数+表格清单；分析类问题 → 六段式研判报告

### 7.3 中文语义检索（无需 Embedding API）
- 实体文本注入中文类型名（如 `supplier 供应商 鑫达原材料...`）
- 滑动窗口中文分词 + 实体类型加权（+15分）
- 即便 DeepSeek 不支持 Embedding，关键词检索依然精准命中

### 7.4 多维风险评分引擎
- 7维度量化打分：延期/库存/财务逾期/合同/质检/设备/物流
- 每实体生成风险分数和等级（严重≥70 / 高风险≥40 / 中等≥20 / 关注）
- 分数和理由在推理日志中透明展示

### 7.5 图谱实时联动
- WebSocket 每阶段推送增量节点+连线
- 3波次展开：检索核心实体 → 遍历上下游 → AI预判虚线
- 节点智能定位（就近邻居中心点）+ 模拟轻推（alpha仅+0.05）

### 7.6 知识图谱本体模型
- 自动生成：系统解析 ERP 结构化数据，自动抽取实体及关系
- 系统定义：本体模型（实体类型、关系类型、属性字段）预先编码，规范数据语义

## 八、技术栈

| 层级 | 技术 | 选型理由 |
|------|------|----------|
| 语言 | Go 1.23+ | 单二进制、M4原生arm64、高性能 |
| AI框架 | Langchaingo (HTTP直连) | 统一LLM+Embedding+结构化输出 |
| 大模型 | DeepSeek (deepseek-chat) | OpenAI兼容协议、高性价比 |
| 向量库 | Chroma (本地进程) | 语义文档知识库、单文件持久化 |
| 图谱库 | BoltDB (Go嵌入) | 关联链路知识库、零端口、Go原生KV |
| 实时通信 | gorilla/websocket | 全量实时推送图谱+聊天+推理日志 |
| 可视化 | D3.js v7 (CDN) | 力导向图谱、缩放拖拽、动画丰富 |
| 前端 | 原生HTML/CSS/JS | 一体化嵌入Go代码，零构建工具 |
