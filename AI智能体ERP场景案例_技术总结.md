# AI智能体ERP场景案例
## 数据ETL / 模型推理 / RAG知识库 / 知识图谱 / 自主研判 一体化案例

---

## 一、项目定位

**轻量化 Palantir 情报研判范式，落地 ERP 供应链智能分析**

| 维度 | 说明 |
|------|------|
| 核心能力 | 多实体自动抽取 → 语义检索（实体+知识库） → 图谱关联 → AI推演预判 → 实时可视化 |
| 技术路线 | Go + DeepSeek + Chroma + BoltDB + D3.js |
| 部署方式 | 单二进制文件，一键启动 |
| 数据规模 | 12类YonSuite实体，240条演示数据，全链路互联 |

## 二、双知识库架构（核心差异化能力）

本项目拥有两套知识库，各司其职、缺一不可：

### 2.1 向量知识库（Chroma — 语义检索层）

> 双集合设计：`erp_entities`（实体文档） + `knowledge_base`（上传知识库文档）

| 维度 | 说明 |
|------|------|
| 存储内容 | ERP 实体描述文本 + 用户上传的知识库文档 + Embedding 向量 + 元数据 |
| 核心作用 | 用户自然语言提问不靠关键词匹配，靠语义相似度同时检索实体和知识库文档 |
| 防幻觉 | LLM 回答时读取原文，保证结论可溯源、不编造 |
| 归属 | RAG 检索、对话回答的底层知识库 |

### 2.2 图谱关系知识库（BoltDB — 关联链路层）

> 属于**结构化关系知识库**，是 Palantir 模式的灵魂

| 维度 | 说明 |
|------|------|
| 存储内容 | 业务关系三元组 `[实体A] —[关系]→ [实体B]`，如 `sup001 —[供货]→ mat001` |
| 核心作用 | 把向量库召回的孤立实体连成完整业务链路，实现多层深度溯源、上下游穿透分析 |
| 可视化 | D3 力导向图谱实时渲染完整业务网络 |
| 归属 | 图谱遍历、关联溯源、可视化渲染的底层知识库 |

### 2.3 双库对比：普通RAG vs 本平台

| 对比维度 | 普通单向量知识库 | 本平台 ERP 复合型知识库 |
|----------|-----------------|------------------------|
| 存储载体 | 仅向量库 | **Chroma 双集合 + BoltDB 图谱库** 多库联动 |
| 存储内容 | 纯文本段落 | 实体描述 + 上传知识库文档 + **实体关联关系** |
| 检索能力 | 仅语义匹配 | 1. 语义召回实体+知识库 2. **自动遍历多层业务链路** 3. AI推演 |
| 可视化 | 无实体链路图谱 | **D3 力导向图谱实时渲染**，分阶段增量展开 |
| 决策能力 | 无推理流程记录 | **完整5步AI决策日志**，展示检索深度、执行步骤 |
| 溯源能力 | 只能引用文档片段 | 可沿图谱链路**多层穿透溯源**，精确定位风险根因 |

## 四、RAG 研判完整查询流程（5步）

```
用户提问
  │
  ▼
┌─────────────────────────────────────────────┐
│ Step 1: 意图识别                             │
│ LLM 识别业务域(供应链/采购/财务/库存)          │
│ + 自动设定检索 TopK、图谱遍历深度              │
└──────────────┬──────────────────────────────┘
               ▼
┌─────────────────────────────────────────────┐
│ Step 2: 向量召回 (Chroma 双集合)              │
│ 用户问题 → 同时检索 erp_entities + knowledge_base │
│ → 召回相关 ERP 实体 + 知识库文档              │
└──────────────┬──────────────────────────────┘
               ▼
┌─────────────────────────────────────────────┐
│ Step 3: 图谱深度遍历 (BoltDB)                 │
│ 以召回实体为起点 → 查询1~3度关联三元组        │
│ → 串联上下游完整业务链路                      │
│ → 实时推送增量图谱到前端 D3 渲染              │
└──────────────┬──────────────────────────────┘
               ▼
┌─────────────────────────────────────────────┐
│ Step 4: 推演预判 (AI预测)                    │
│ AI 分析现有链路 → 预测下一步待核查实体         │
│ → 生成虚线预判链路推送到图谱                  │
│ → 自动核查目标实体，输出核查结论              │
└──────────────┬──────────────────────────────┘
               ▼
┌─────────────────────────────────────────────┐
│ Step 5: 生成研判报告 (LLM整合)                │
│ 整合向量文档 + 知识库 + 图谱关联链路           │
│ → 弹出"正在生成研判报告..."提示窗              │
│ → 输出带溯源、数据举证的结构化回答             │
└─────────────────────────────────────────────┘
```

每一步推理动作实时记录 **DecisionTrace**，通过 WebSocket 推送前端决策日志面板。

## 五、核心业务执行链路

### 第一层：数据接入层（三模式实体生成入口）

| 模式 | 说明 | 流程 |
|------|------|------|
| API 同步 | 读取本地 Mock JSON 文件 | 读取文件 → 解析实体字段 → 自动建立三元组关系 |
| 批量同步 | 一键同步全部12类（清除旧数据后重新加载） | 清除旧实体 → 12类×20条 → 240条全链路数据 |
| 手动录入 | 页面表单输入实体 + 自定义属性 + 自选关联关系 | 生成实体ID → 存入BoltDB → 同步Chroma向量 |
| 知识库上传 | 上传 txt/md/json/csv 文件 | 向量化 → 存入 Chroma knowledge_base + BoltDB 日志 |

### 第二层：多存储持久化层

| 分支 | 存储 | 流程 |
|------|------|------|
| 向量分支 | Chroma | 实体文本 + 知识库文档 → Embedding → 分别存入 erp_entities / knowledge_base |
| 图谱分支 | BoltDB | 解析实体外键关联 → 生成三元组 → 写入图谱库 |
| 日志分支 | BoltDB | 操作日志、聊天记录、知识库文档元数据持久化 |

同步通过 WebSocket 推送新增实体节点、连线至前端 D3 图谱实时渲染。

### 第三层：AI 研判推理层（5步决策引擎）

5步完整研判流程（详见第四章），每一步携带推理理由和 AI 下一步计划。
研判报告生成时弹出提示窗，报告返回后自动关闭。

### 第四层：实时可视化展示层

三列布局，亮色卡片式主题，实时联动：
- **左列**：对话 Chat 窗口，Markdown 报告渲染（表格/列表/代码块/粗体）
- **中列**：D3 力导向图谱，分阶段增量展开 + 自动缩放适配
- **右列**：API同步/手动录入/知识库面板 + AI 推理日志（显示当前步、AI 计划、多分支路径）

## 七、核心技术亮点

### 7.1 多知识库联动检索
- Chroma 双集合（实体 + 知识库）语义召回 → BoltDB 链路深挖 → LLM 整合多源数据
- 不是简单文档问答，而是「语义理解 + 结构化推理」的复合智能

### 7.2 智能意图识别 + 自适应回答
- LLM 自动判断查询类型（列举/分析/风险）
- 自适应调整检索参数（TopK=3~20、遍历深度=1~3）
- 列举类问题 → 总数+表格清单；分析类问题 → 五段式研判报告

### 7.3 中文语义检索（无需 Embedding API）
- 实体文本注入中文类型名（如 `supplier 供应商 鑫达原材料...`）
- 滑动窗口中文分词 + 实体类型加权（+15分）
- 同时检索知识库文档
- 即便 DeepSeek 不支持 Embedding，关键词检索依然精准命中

### 7.4 图谱实时联动
- WebSocket 每阶段推送增量节点+连线
- 3波次展开：检索核心实体 → 遍历上下游 → AI预判虚线
- 节点智能定位（就近邻居中心点）+ 模拟轻推

### 7.5 知识图谱本体模型
- 自动生成：系统解析 ERP 结构化数据，自动抽取实体及关系
- 系统定义：本体模型（实体类型、关系类型、属性字段）预先编码，规范数据语义

### 7.6 数据同步非累加
- 批量同步前自动清除旧实体和三元组，确保数据一致性
- 独立知识库文档查询，不受日志数量限制

## 八、技术栈

| 层级 | 技术 | 选型理由 |
|------|------|----------|
| 语言 | Go 1.23+ | 单二进制、arm64原生、高性能 |
| 大模型 | DeepSeek (deepseek-chat) | OpenAI兼容协议、高性价比 |
| 向量库 | Chroma (本地) | 双集合（实体+知识库）、语义检索 |
| 图谱库 | BoltDB (Go嵌入) | 关联链路知识库、零端口、Go原生KV |
| 实时通信 | gorilla/websocket | 全量实时推送图谱+聊天+推理日志 |
| 可视化 | D3.js v7 + Chart.js | 力导向图谱、图表渲染 |
| 前端 | 原生HTML/CSS/JS | 亮色卡片式主题、嵌入Go代码 |
